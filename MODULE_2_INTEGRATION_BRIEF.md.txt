### Calculation logic (move verbatim, mirror the Excel exactly)
| Item | Formula |
|---|---|
| Reclassification | `emp_no` starting with `'300'` → SUB |
| Wage / Allowance | From `003_Cleaned_Salary_Report.xlsx` (col F / G+I+J+L) |
| OT cap per month | `Wage × OT_mult × (OT_cap / (1 − OT_cap))` |

| KPI bonus annual | `(Wage + Allowance) × KPI_months × FY_service_days / 365`, spread evenly across 12 |

| BONUS file split | KPI 40% + PBO 60% of (PER+SUB KPI total); Fixed 100% of (PER+SUB Fixed total) |
| Service days FY | Days from `max(joined_date, 2026-07-01)` through `2027-06-30`, capped at 365 |
| Service days Cal26 | Days from `max(joined_date, 2026-01-01)` through `2026-12-31`, capped at 365 |
| Intern monthly wage | `intern_daily × FY27_total_working_days / 12` (Mon–Fri minus holidays) |
| DUM001/DUM002 | `cap × wage_pct` (wage), `cap × (1−wage_pct)` (allowance) |

## 3. ⚠️ Salary data architecture (the key decision)

### The problem
The Manpower Budget computes per-employee budget lines from individual salary records. The Interactive Table page displays those lines. The host app's database is **ephemeral on Streamlit Cloud** (every redeploy resets SQLite — see B2 in the main handoff).

### Decision: persist in SQLite, gate at the app layer with capabilities

**Architecture:**
1. New table `salary_master` keyed by `(emp_no, fy)`. Holds wage, allowance, OT potential, etc.
2. New table `salary_upload_log` for audit (who uploaded, when, hash of source file).
3. Add 6 new capabilities: `salary.view`, `salary.upload`, `salary.export`, `budget.access`, `budget.edit`, `budget.export`.
4. Every salary read goes through `lib/salary_data.py` which checks `auth.has_capability(username, "salary.view")` first. **Pages never SELECT * FROM salary_master directly.**
5. Aggregate views (totals, charts) require no salary cap — only the raw rows / individual line items do.
6. Display "Salary data last uploaded: 2026-05-08 by hr_admin" prominently on every Module 2 page. Warn if data older than 45 days.

**Trade-off acknowledged:** Streamlit Cloud's SQLite resets on every redeploy, so salary data must be re-uploaded by Finance after each redeploy. Mitigations:
- A clear "🔄 Re-upload required" banner appears when `salary_master` is empty
- Upload is one click (Finance Manager has `salary.upload`)
- Long-term fix: Postgres migration (B2 in main handoff)

**Alternatives evaluated:**

| Option | Verdict | Reason |
|---|---|---|
| Session-only (each user uploads at login) | ❌ rejected | Other Finance Managers can't see budget; UX painful |
| Encrypted file committed to git | ❌ rejected | Data in git history; encryption-at-rest doesn't help if app can decrypt |
| External API (Google Sheets / Airtable) | ❌ rejected | New blast radius, no clear gain over SQLite + caps |
| **SQLite + capability gate** | ✅ chosen | Aligns with v11.4 model; clear migration path |
| Postgres (Supabase free tier) | 🎯 future | Right answer when B2 lands |

### Capability design

| Capability | EN description | TH description | Default roles |
|---|---|---|---|
| `salary.view` | Read individual salary records | ดูข้อมูลเงินเดือนรายบุคคล | finance, admin, super_admin |
| `salary.upload` | Upload salary master file | อัปโหลดข้อมูลเงินเดือน | finance, admin, super_admin |
| `salary.export` | Include salary in downloaded files | รวมข้อมูลเงินเดือนในไฟล์ส่งออก | finance, super_admin |
| `budget.access` | See Budget Module on hub | เข้าถึงโมดูลงบประมาณกำลังคน | finance, admin, super_admin |
| `budget.edit` | Edit assumptions, toggles, holidays | แก้ไขสมมติฐานงบประมาณ | finance, super_admin |
| `budget.export` | Download deliverable budget files | ดาวน์โหลดไฟล์งบประมาณ | finance, admin, super_admin |

**Rationale:**
- Manager and Supervisor roles get NO budget/salary access — protects sensitive comp data
- Admin can edit budget assumptions but cannot export salary in the clear (`salary.export` denied) — separation of duties
- Finance gets full access — they own the process
- Super Admin gets everything — escape hatch

### Audit trail

Every salary read by a user gets logged to `salary_access_log` with `(username, accessed_at, action, emp_no_count)`. One row per page-load that touches salary, not per individual record.

## 4. Database additions

Add to a new `lib/manpower_budget_migration.py` following the `rbac_migration.py` pattern. All `CREATE TABLE IF NOT EXISTS`, additive, idempotent.

```sql
CREATE TABLE IF NOT EXISTS salary_master (
    emp_no             TEXT NOT NULL,
    fy                 TEXT NOT NULL,
    wage               REAL NOT NULL,
    allowance          REAL DEFAULT 0,
    ot_potential       REAL DEFAULT 0,
    feb_actual_ot      REAL DEFAULT 0,
    notes              TEXT,
    uploaded_at        TEXT DEFAULT CURRENT_TIMESTAMP,
    uploaded_by        TEXT,
    PRIMARY KEY (emp_no, fy),
    FOREIGN KEY (emp_no) REFERENCES employees(emp_no)
);

CREATE TABLE IF NOT EXISTS salary_upload_log (
    id                 INTEGER PRIMARY KEY AUTOINCREMENT,
    fy                 TEXT NOT NULL,
    rows_inserted      INTEGER,
    rows_updated       INTEGER,
    source_filename    TEXT,
    source_sha256      TEXT,
    uploaded_by        TEXT NOT NULL,
    uploaded_at        TEXT DEFAULT CURRENT_TIMESTAMP
);

CREATE TABLE IF NOT EXISTS salary_access_log (
    id                 INTEGER PRIMARY KEY AUTOINCREMENT,
    username           TEXT NOT NULL,
    action             TEXT NOT NULL,
    emp_no_count       INTEGER,
    accessed_at        TEXT DEFAULT CURRENT_TIMESTAMP
);
CREATE INDEX IF NOT EXISTS idx_salary_access_user ON salary_access_log(username, accessed_at);

CREATE TABLE IF NOT EXISTS manpower_budget_assumptions (
    fy                 TEXT PRIMARY KEY,
    ot_cap_pct         REAL,
    ot_mult            REAL,
    kpi_months         REAL,
    fixed_l3           REAL,
    fixed_l12          REAL,
    bonus_kpi_pct      REAL,
    bonus_pbo_pct      REAL,
    tog_tem            INTEGER DEFAULT 0,
    intern_daily       REAL,
    tog_dum001         INTEGER DEFAULT 0,
    dum001_cap         REAL,
    dum001_wage_pct    REAL,
    tog_dum002         INTEGER DEFAULT 0,
    dum002_cap         REAL,
    dum002_wage_pct    REAL,
    updated_at         TEXT DEFAULT CURRENT_TIMESTAMP,
    updated_by         TEXT
);

CREATE TABLE IF NOT EXISTS manpower_budget_overrides (
    fy                 TEXT NOT NULL,
    emp_no             TEXT NOT NULL,
    field              TEXT NOT NULL,
    value              REAL,
    reason             TEXT,
    updated_at         TEXT DEFAULT CURRENT_TIMESTAMP,
    updated_by         TEXT,
    PRIMARY KEY (fy, emp_no, field)
);

CREATE TABLE IF NOT EXISTS manpower_budget_audit (
    id                 INTEGER PRIMARY KEY AUTOINCREMENT,
    fy                 TEXT NOT NULL,
    action             TEXT NOT NULL,
    detail             TEXT,
    username           TEXT NOT NULL,
    ts                 TEXT DEFAULT CURRENT_TIMESTAMP
);
```

**Migration timing:** call `apply_module2_migration()` AFTER `apply_rbac_migration()` in `db.init_db()` (same pattern — outside the main `with cursor()` block, in its own connection, to avoid the lock contention bug fixed in v11.4 hotfix).

```python
# v11.5 backfill: add Module 2 caps to existing role_capabilities entries
NEW_V11_5_CAPS = {
    "finance":     ["salary.view", "salary.upload", "salary.export",
                    "budget.access", "budget.edit", "budget.export"],
    "admin":       ["salary.view", "salary.upload",
                    "budget.access", "budget.export"],
    "super_admin": ["salary.view", "salary.upload", "salary.export",
                    "budget.access", "budget.edit", "budget.export"],
}
for role, caps in NEW_V11_5_CAPS.items():
    for cap in caps:
        conn.execute("""INSERT OR IGNORE INTO role_capabilities
                        (role_key, capability_key) VALUES (?, ?)""",
                     (role, cap))
```

`INSERT OR IGNORE` won't overwrite Super-Admin-edited rows.

## 10. Risks and mitigations

| Risk | Mitigation |
|---|---|
| Salary data lost on Cloud redeploy | Banner + one-click re-upload; long-term: Postgres migration (B2) |
| Backfill conflicts with Super-Admin matrix edits | `INSERT OR IGNORE` won't overwrite. Ship backfill once, then remove from seed (one-shot pattern). |
| Numbers don't match Excel | Side-by-side comparison; standalone is already within < 1,000 THB |
| Salary upload UX confusing | Clear empty-state with arrow at upload button; "last uploaded by X" always visible |
| Audit log fills disk | Periodic prune (last 90 days). Streamlit Cloud's ephemeral disk resets anyway |
| Override race condition | Keyed by (fy, emp_no, field) — last write wins. Add `updated_at` to UI |
| Page-letter scheme exhaustion | Use one page with tabs (Section 6) |
